## 与卡牌系统同步口径（批次 C）
来源：卡牌系统文档草案 v0.2，revision 557。以下内容为当前系统案可同步口径；GDD、旧 PRD、会议纪要等来源材料不直接等于最终规则，迁入本系统案前需经过保留 / 改写 / 废弃 / 待确认处理。
### 关卡 / 建筑系统定位
- 建筑不是普通贴图，而是由功能牌节点构成的可交互结构。
- 塔楼关卡系统负责定义本关建筑结构、功能牌节点分布、节点状态、关卡目标和坍塌表现。
- 卡牌系统负责基础牌出牌、牌型判定、功能牌触发和坍塌值计算；塔楼关卡系统接收这些结果并更新建筑状态。
### 本关功能牌节点管理
关卡 / 建筑系统需要管理以下节点数据：
- 本关功能牌节点列表。
- 节点摆放位置。
- 节点解锁条件。
- 节点当前是否可被抽出。
- 节点抽出后的建筑表现。
- 节点是否属于等级一功能牌 / 等级一节点已拍板规则。
### 节点触发与抽出流程
1. 玩家使用基础扑克牌出牌。
1. 卡牌系统判定本次出牌是否满足某个建筑功能牌节点的解锁条件。
1. 若满足条件，功能牌从建筑节点中抽出。
1. 抽出的功能牌进入玩家可操作区 / 手牌，由玩家后续主动使用。
1. 建筑节点进入已抽出 / 缺失 / 破坏状态。
1. 建筑表现根据节点配置播放松动、裂开、抽出、坍塌等反馈。
### 坍塌值目标与结算接口
- 当前关卡拥有目标坍塌值 Target_level。总控已确认首轮模拟基线：Level 1 / T1 = 900，Level 2 / T2 = 1200；该基线不是最终正式难度值，首轮模拟后仍可调参。
- 玩家基础出牌造成的坍塌总值为 S_total。
- 功能牌节点抽出造成的额外坍塌值为 E_total。
- 当前关卡累计总坍塌值使用卡牌系统口径：
```
Collapse_total = S_total + E_total
```
- 已拍板：E_total 进入首版数值模拟与程序实现。
- 已拍板：等级一 100% 坍塌属于节点抽出 / E_total 阶段，必须先于 Collapse_total 更新后的胜利判断。
- 仍需程序 / 配置确认：字段命名、数值参数、表现资源和失败 / 重置命名。
- 当 Collapse_total >= Target_level 时，当前关卡可进入成功结算。
- E_total 已进入首版数值模拟与程序实现；关卡结算按 Collapse_total = S_total + E_total 对齐卡牌系统 revision 557。
### 建筑表现需求
节点抽出与坍塌值变化需要驱动建筑表现，至少包括以下表现方向：
- 节点满足条件时的可抽出提示。
- 功能牌从建筑中抽出的动效或状态变化。
- 节点抽出后的缺口、裂缝、松动或结构变化。
- 关卡累计坍塌值接近目标时的建筑阶段性反馈。
- 达成 Target_level 后的建筑坍塌 / 通关表现。
### 等级一节点规则（r557 已拍板）
等级一功能牌 / 等级一节点规则已由 Q-CARD-004 与卡牌系统 revision 557 拍板：
- 等级一 100% 坍塌进入当前规则。
- 100% 坍塌发生在节点抽出 / E_total 阶段，先于 Collapse_total 更新后的胜利判断。
- 通关后的降级、继承、奖励和功能牌池更新由 RewardSystem / 关卡结算处理。
- 待确认项仅保留降级目标、继承字段、表现资源、配置字段细节、是否需要特殊胜利节点 UI 表现。
### 与数据配置的接口
塔楼关卡系统至少需要从数据配置读取：
- 功能牌节点的角色 node_role。
- 节点解锁条件 unlock_condition_type / unlock_condition_value。
- 是否等级一胜利节点 is_level_one_victory_node。
- 抽出额外坍塌值 extract_score_type / extract_score_value。
- 抽出建筑表现 on_extract_building_effect。
- 节点视觉提示 visual_hint。
